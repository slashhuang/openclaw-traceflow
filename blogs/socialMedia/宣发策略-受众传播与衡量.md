## 一、受众分析（按平台）

不同场子，读者**动机、耐心、信任方式**都不一样；同一篇稿子「一稿多发」可以，但**首屏钩子、长度、配图、评论区话术**要按场子微调。

| 平台 | 典型受众 | 来这想解决什么 | 对 TraceFlow 最吃哪一口 | 内容形态建议 |
|------|-----------|----------------|-------------------------|--------------|
| **微信公众号** | 关注你的熟人、同行、行业群转发进来的读者；偏「订阅 + 社交推荐」 | 想**一口气读完故事 + 观点 + 技术梗**；信任作者本人 | **你是谁（重度用户、研发场景、git worktree/多人协作）** + **可观测/黑盒变白盒** + **踩坑故事** | 长文 2000～4000 字；**首段 3 屏内必须有痛点**；结尾固定 **GitHub + Star**；可配 1～2 张产品截图 |
| **知乎** | 搜索进来的人、关注「Agent / 后端 / 可观测」话题的用户 | 要**可验证、可收藏、能当工具书** | **与默认后台差异**、**黑盒/上下文**、**ROADMAP 诚实** | 专栏 or 回答：**小标题 + 引用**；避免纯软文腔；文末 **Star** 比「求赞」更自然 |
| **小红书** | 算法推荐 + 搜索；偏年轻、视觉敏感；技术内容也吃「**情绪 + 短句**」 | 想**快速共鸣、收藏、转发给同事** | **标题大字痛点**、**Skill 黑盒**、**开发花絮（搜狗+Claude Code+Cursor+阿里云）** | **一篇一主题**；封面图 + 短段落；话题标签；**评论置顶 GitHub** |

**标题口径（建议三端一致）**：优先戳 **黑盒 / 盲盒 / 上下文可见性 /「模型输出与会话依据对不齐」**（可与「记忆幻觉」无力感类比，**不夸大产品能「治幻觉」**）；**避免**用「钱、老板、算账」当标题钩子——稳重、像真排障，不像标题党。

**交叉人群**：三边都可能有「带团队 / 一人公司 / 研发负责人」——公众号偏**关系与故事**，知乎偏**说理与检索**，小红书偏**情绪与记忆点**。

## 二、媒体传播路径的演变（简版，便于你心里有一张图）

理解「内容怎么流动」，才好在**首发、二发、沉淀**上分配精力。

| 阶段 | 典型路径（近年共性） | 对开源项目的含义 |
|------|----------------------|------------------|
| **私域/订阅** | 公众号、朋友圈、社群 → 第一波信任转化 | 适合讲**完整故事 + 身份背书**，转化 **Star / 加群讨论** |
| **公域算法** | 小红书、知乎推荐流、话题聚合 | 标题和**前 15 秒/首段**决定生死；适合**单点痛点**反复发不同切口 |
| **搜索长尾** | 知乎、微信搜一搜、百度对知乎/文章的收录 | **问题标题**（OpenClaw 可观测、Skill 黑盒、git worktree）带来**长期流量** |
| **开发者圈层** | GitHub Trending、推特/X、V2EX、即刻、技术 Newsletter | **英文一句 + 截图**往往比长文中译更有效；国内可同步到**掘金/少数派**等（按需） |

**建议节奏（示例）**：  
1）**知乎**：先发一个「问题驱动」长回答（易收录）→  
2）**公众号**：同一素材扩写故事与身份 →  
3）**小红书**：拆成 3～5 条笔记，每条一个钩子 →  
4）所有出口统一指向 **GitHub 仓库**，避免链接分散。

## 三、效果衡量（别只盯阅读量）

开源宣发**没有单一金指标**；建议分层看，避免「虚荣指标」骗自己。

| 层级 | 指标（示例） | 说明 |
|------|----------------|------|
| **北极星（项目）** | GitHub **Stars** 增速、**Fork**、**Issue/PR** 质量 | Star 是弱承诺，但**可持续**；有质量的 Issue 比十万阅读更有用 |
| **传播** | 公众号阅读、在看、分享；知乎阅读、收藏、赞同；小红书曝光、收藏、评论 | 各平台后台可见；**收藏/转发**往往比点赞更接近「真有用」 |
| **转化** | 从文章到 GitHub 的点击（可用短链或 UTM，若你愿意折腾） | 粗粒度即可：某篇发出后 **48h 内 Star 增量** 对比平日 |
| **定性** | 评论里是否出现「Skill 看不清」「上下文对不齐」等 | **话术与真实痛点对齐**，就说明叙事对了 |

**建议**：发完后记一条「**日期 + 渠道 + 主标题 + 48h Star 变化**」表格，发三次就能摸到自己的**有效渠道**。

## 六、你可直接执行的「最小发布包」

1. **知乎**：1 篇长回答（第二节正文）+ 若干条「想法」分流（第三节）；优先用「适合回答的问题」里的标题去**答具体问题**。  
2. **公众号**：1 篇（见 `wechat-article-openclaw-traceflow.md`），定时晚 8～10 点或你读者习惯时段。  
3. **小红书**：同一周按 **A → B → F** 或 **A → F → C** 发 3 条（见文内「建议顺序」）。  
4. **衡量**：记录发后 48h **GitHub Star** 增量 + 各平台**收藏数**。  

知乎与公众号、小红书已**分文件优化**，一般无需再「公众号压成知乎版」；若需**单条 500 字纯小红书成稿**，可再单独拆文件。
