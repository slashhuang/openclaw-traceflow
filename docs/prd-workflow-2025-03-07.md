# 📋 PRD 与需求文档规范（Requirement-Driven Development）

本仓库以「需求先行」方式迭代：功能或结构类变更须先有已批准的需求文档，再动手实现。

## 📁 docs 下文档命名规范

| 类型 | 命名 | 位置 | 说明 |
|------|------|------|------|
| **产品需求（PRD）** | `prd-<英文主题>-YYYY-MM-DD.md` | `docs/` | 产品级需求说明，小写+日期，如 `prd-feishu-github-sync-2025-03-07.md`、`prd-workflow-2025-03-07.md` |
| **细粒度需求（可选）** | `req-<short-id>-<lowercase-dash-title>.md` | `docs/req/` | 从 PRD 拆出的单条可交付需求，如 `req-001-add-bitable-sync.md` |
| **技术/约定文档** | 其它命名（如 `SKILL_*.md`） | `docs/` | 非需求类说明，如技能约定、运维说明等 |

**约定**：凡属「产品需求」的文档，**一律小写并以 `prd-` 开头、带日期 `-YYYY-MM-DD`**，便于检索与规范化迭代。

---

## ✅ 何时需要写 PRD

- 新增功能或集成
- 重构核心逻辑或架构
- 变更 API、数据模型或 CLI 接口
- 任何影响正确性、安全或用户可见行为的改动

---

## 🧾 PRD 建议结构

| 区块 | 用途 |
|------|------|
| `## Context` | 背景与动机 — 痛点、触发点或机会 |
| `## Goal` | 目标 — 成功长什么样（用户或系统视角） |
| `## Acceptance Criteria` | 可验收条件 — 可测试、具体（Gherkin 风格更佳） |
| `## Out of Scope` | 明确边界 — 本次**不**包含的内容 |
| `## Stakeholders` | 提出方、审批方、实现方 |
| `## Status` | `draft` → `review` → `approved` → `implemented` → `verified` |

细粒度需求（`docs/req/`）可沿用上述结构，并注明关联的 PRD 文件名。

---

## 🔄 评审与合并流程

1. 作者在针对 `main` 的 PR 中新增/修改 `docs/prd-*-YYYY-MM-DD.md`（或 `docs/req/req-*.md`）。
2. 至少两位评审（含一位领域负责人）给出 `✅ approved` 或修改意见。
3. 合并后即可据此开发；实现 PR 须在描述中关联需求文档（如 `Closes: docs/prd-feishu-github-sync-2025-03-07.md`）。

---

## 🌐 分支约定（可选）

- `main`：稳定、可发布
- `dev`：集成用（从已合并的 feature 分支自动合并）
- `feat/req-XXX` 或 `feat/PRD-<主题>`：实现分支，与需求 ID/PRD 对应

> 💡 需求合并后再拉实现分支，例如：`git checkout -b feat/PRD-feishu-github-sync`

## 需求三：飞书指令 → PRD 先行 + Git Worktree → GitHub PR 交付

当**需求来源为飞书上的用户指令**且**涉及本仓库的修改**时，执行方（如阿布/Agent）必须遵循「先 PRD 确认、再通过 Git worktree 实现、交付物为 GitHub PR」的完整流程。技术细节（worktree 命令、分支命名、判断表）见 `docs/PR-WORKFLOW.md`。

### Context

- 用户在**飞书**上向阿布（或同类 Agent）发出指令（如「帮我加一个 XXX 功能」「改一下配置」）。若指令会**改动本仓库（claw-family）的代码或配置**，若不统一规范，容易出现：直接在主工作区改、无 PR；需求理解偏差导致返工；或未先写 PRD 就动手，范围蔓延。
- 诉求明确为：
  1. **飞书上的指令**若涉及本仓库修改 → **必须**通过 **Git worktree** 实现，**不得**在主工作区直接改。
  2. **交付物**为 **GitHub 上的 Pull Request**（而非「改完告诉你」或仅本地提交）。
  3. **需求不清晰时**，执行方**必须主动向用户询问**，不猜测实施。
  4. **功能扩展类**先写 PRD 并得到用户确认后再实施；**修 bug（fix）**可不写 PRD，直接实施。
  5. **只要涉及本仓库的修改**，一律通过 PR 提出，无例外（紧急 hotfix 可事后补 PR，见既有约定）。

### Goal

- **用户视角**：在飞书里提需求 → 若涉及本仓库，会先收到需求澄清（必要时）；若为**功能扩展**则先收到 **PRD 的 PR**（单独一个 PR）供确认，用户在飞书内明确说「确认」「同意」「可以」后 PRD PR 合并，用户再指令「基于该 PRD 实施」→ 执行方提**实现 PR**；若为**修 bug** 则直接走 worktree + PR → 最终收到 **GitHub PR 链接**，且飞书回复按 AC6 包含是否涉及本仓库、是否走 PRD、worktree 路径、PR 链接（不涉及时仅说明「本次不涉及本仓库修改」）。
- **执行方视角**：收到飞书指令 → 判断是否涉及本仓库修改 → 若涉及，则（1）需求不清则先询问；（2）若为功能扩展则先为 PRD 提单独 PR 并展示给用户，用户飞书明确同意后等 PRD PR 合并，用户再发「基于该 PRD 实施」类指令时，为实现创建 worktree 并提实现 PR；若为修 bug 则不写 PRD，直接 worktree + PR；（3）（4）推送后创建 PR，并在飞书中按 AC6 回复。

### Acceptance Criteria

- **AC1（需求澄清）**  
  - 当指令意图、范围、验收标准等**不明确**时，执行方在动手前**必须**向用户提问澄清（如：具体要改哪些文件、期望行为、是否先写 PRD）。  
  - 仅在得到足够清晰的信息或用户明确「先按你的理解写 PRD 我再确认」后，才进入 PRD 编写或实施。

- **AC2（PRD 先行）**  
  - **修 bug（fix）**：不需先写 PRD，可直接在 worktree 内实现并提 PR。  
  - **功能扩展（非 fix）**：凡新功能、重构、结构/行为类变更等，**必须先**在 `docs/` 下编写或更新 PRD（或 `docs/req/` 下细粒度需求）。PRD 文档本身也走 worktree + PR，且为**单独一个 PR**：先为「PRD 文档」创建分支与 worktree，提交仅含 PRD 的 PR 并将 PRD 内容/链接展示给用户；**必须用户在飞书里明确说「确认」「同意」「可以」等之后**，PRD 的 PR **须先合并到 main**；合并后，由用户另行在飞书指令（如「基于该 PRD 实施」），执行方再创建**第二个** worktree 与分支，进行代码/配置实现并提实现 PR。未在飞书内明确同意前，不进入实现阶段；PRD 未合并前，不开始实现。  
  - 边界不清时以「是否为修 bug / fix」为准：用户明确说是修 bug 则不走 PRD；否则按功能扩展处理，走 PRD 流程。

- **AC3（必须使用 Git worktree）**  
  - 凡涉及本仓库代码或配置修改的实施，**必须**为该需求创建独立分支，并用 `git worktree add` 创建独立工作目录；**仅在该 worktree 内**进行编辑、提交、推送。  
  - 主工作区（用户打开的本仓库目录）保持 `main`，不因该需求而切换分支。  
  - 分支命名、worktree 位置等遵循 `docs/PR-WORKFLOW.md`。

- **AC4（交付物为 GitHub PR）**  
  - 在 worktree 内完成开发并 push 后，**必须**在 GitHub 上创建指向 `main` 的 Pull Request。  
  - 执行方将 **PR 链接**作为对用户的**主要交付物**回复（可附带简短说明）；不以「已改好」或「已提交到某分支」代替 PR 链接。  
  - PR 描述中须能看出对应需求；若存在 PRD/req 文档，PR 中须有关联引用（如 `Closes: docs/prd-xxx.md`）。

- **AC5（涉及本仓库即走 PR）**  
  - **只要**修改涉及本仓库的代码、配置、文档、目录结构等（以 `docs/PR-WORKFLOW.md` 的「判断标准」为准），**一律**通过 PR 提出，无「只改一点就不提 PR」的默认例外。  
  - **例外**：仅限**纯文案/拼写类 typo**（不改变逻辑、不新增文件）可视为既有约定的「文档 typo」例外；紧急 hotfix 可事后补 PR。**其余修改**（含修 bug）一律走 worktree + PR；修 bug 不写 PRD，但必须提 PR。例外情况应在 PR 或说明中注明。

- **AC6（飞书回复格式）**  
  - **当涉及本仓库时**，执行方在飞书中对用户的回复**必须**包含以下信息（在流程推进到相应阶段时给出）：  
    - **是否涉及本仓库**：本次指令是否会修改 claw-family 的代码/配置；  
    - **是否先写 PRD**：若为功能扩展则说明「先写 PRD 供确认」；若为修 bug 则说明「按 fix 流程，不写 PRD」；  
    - **worktree 路径**：若已创建 worktree，给出其路径（如 `../claw-family--feat-xxx`）；  
    - **PR 链接**：完成开发并创建 PR 后，必须将 GitHub PR 链接回复给用户。  
  - **当不涉及本仓库时**，只需明确说明「本次不涉及本仓库修改」，其余三项可省略或标「不适用」。  
  - 便于用户在飞书内即可追踪：是否动仓库、是否走 PRD、在哪开发、最终 PR 在哪。

### Out of Scope

- 不规定**非本仓库**的修改流程（如只改其他项目、只做数据分析）：本需求仅约束「涉及 claw-family 仓库」的飞书指令。
- 技术细节（worktree 命令、分支命名、判断表）以 `docs/PR-WORKFLOW.md` 为准；本需求侧重「飞书入口 + PRD 先行 + 交付物 = PR」的行为与验收。
- 不规定飞书机器人或 OpenClaw 的底层实现方式（如具体 API、会话存储），仅规定「在飞书场景下，执行方应表现出的流程与交付行为」。

### 依赖与前置条件

- 本需求假设**执行方具备**在可访问本仓库（clone）的环境内执行 `git worktree`、`git push` 及在 GitHub 上创建 Pull Request 的能力；具体环境与权限由部署方保证。

### 最佳实践与扩充（执行方可依此细化）

- **需求澄清**：若指令较泛（如「优化一下启动脚本」），主动列出可选项或请用户确认范围（例如：是加检查、改日志、还是改参数？），避免做多余或偏离的改动。  
- **PRD 粒度**：**修 bug（fix）不写 PRD**；**功能扩展（非 fix）均须先 PRD 并获确认后再实施**；PRD 本身单独一个 PR，用户飞书明确同意后 **PRD PR 须先合并**，用户再指令「基于该 PRD 实施」时执行方再开实现 PR（与 AC2 一致）。  
- **typo 与修 bug**：仅限**纯文案/拼写类 typo**（不改变逻辑、不新增文件）可视为「文档 typo」例外；其余修改（含修 bug）一律走 worktree + PR；修 bug 不写 PRD，但必须提 PR。  
- **一次指令一 PR**：功能扩展时 PRD 与实现各一个 PR；一次飞书对话中的多个小改动可合并为一个 PR，但需在 PR 描述中分点说明；若用户明确拆成多个需求，则对应多个分支与 PR。  
- **回复格式**：飞书回复须满足 AC6；涉及本仓库时须包含「是否涉及本仓库、是否先写 PRD、worktree 路径、PR 链接」，不涉及时只需说明「本次不涉及本仓库修改」。  
- **与现有文档衔接**：执行方应同时遵守本文档（评审与合并流程、需求三）、`docs/PR-WORKFLOW.md`（worktree、分支、判断标准与命令示例）；本需求三是在「飞书指令」这一入口上叠加「PRD 先行 + 交付物 = PR」的约束。

| 项目 | 说明 |
|------|------|

| **审批方** | 仓库维护者 / 家庭成员 |

| **Status** | Draft → 经你确认后可标为 approved，再据此实施或写入 workspace-defaults/BOOT.md、SOUL.md 等 Agent 指令 |
