# PRD：飞书触发 GitHub 主分支同步与更新

**文档类型**：产品需求说明（PRD）  
**状态**：Draft  
**最后更新**：2026-03-07  

---

## 一、背景与目标

### 1.1 背景

- 本仓库（claw-family）在云端机器上以 PM2 方式运行 OpenClaw Gateway，对接飞书等通道。
- 代码与配置的迭代在本地或 GitHub 主分支进行，希望**云端运行实例能与 GitHub 主分支保持联动**：有变更时自动拉取、按需更新并重启，无需人工 SSH 登录执行。
- 希望**以飞书聊天为触发入口**：用户与飞书里的 Agent 交互时，可作为「有人在用」的信号，进而驱动云端执行「拉取 GitHub → 检测变更 → 更新/重启」的流程。

### 1.2 目标

- **主流程**：用户与飞书聊天 → 云端收到聊天/活跃信号 → 云端主动拉取 GitHub 主分支 → 若发现变更则执行更新（必要时执行重启）。
- **成功标准**：推送代码到 GitHub 主分支后，通过飞书侧的一次交互即可间接触发云端拉取与更新，在约定条件下自动完成更新或重启，无需登录服务器手工操作。

## 二、角色与场景

| 角色 | 说明 |
|------|------|
| **用户** | 在飞书中与 Agent（如阿布）聊天的家庭成员（如 jojo、slashhuang） |
| **云端机器** | 部署并运行 OpenClaw Gateway（PM2）的服务器 |
| **GitHub 仓库** | 本仓库（claw-family）的 main 分支，作为代码与配置的真实来源 |

### 典型场景

1. **日常聊天触发检测**  
   用户在飞书发消息 → 云端收到请求/事件 → 触发一次「拉取 GitHub main → 比较是否有变更」；若无变更则仅记录日志，不重启。

2. **有变更时更新**  
   拉取后发现 main 有新的 commit → 执行更新流程（见下），必要时重启 Gateway 容器。

3. **需重启时**  
   当检测到影响运行的变更（如 `config/`、依赖或启动脚本等）时，在更新后执行重启（如 `pm2 restart claw-gateway`）。

## 三、流程说明

### 3.1 端到端流程（概览）

```
[用户] 飞书发消息
    ↓
[飞书 / OpenClaw] 请求到达云端 Gateway
    ↓
[云端] 收到请求 → 触发「同步检查」逻辑（见 3.2）
    ↓
[云端] git fetch + 比较 main 与当前 HEAD
    ↓
  有变更？
    ├─ 否 → 记录日志，结束
    └─ 是 → 执行更新（git pull / 或 pull + rebuild）
              ↓
         需要重启？
              ├─ 是 → pm2 restart claw-gateway
              └─ 否 → 仅更新文件，结束
```

### 3.2 「同步检查」触发方式（可选方案）

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A. 请求侧钩子** | 在 Gateway 处理飞书请求的入口（或中间件）中，在收到飞书消息时调用「同步检查」逻辑（可节流/防抖） | 与现有架构贴合，无需额外服务 | 需在 OpenClaw 或本仓库封装层增加钩子；需避免阻塞主请求 |
| **B. 独立 Webhook** | 飞书/第三方在「收到消息」时调用云端一个 HTTP 接口，该接口仅负责触发同步检查 | 职责清晰，易限流与监控 | 需配置飞书 Outgoing Webhook 或自建转发，多一层配置 |
| **C. 定时 + 心跳** | 云端定时（如每 5 分钟）检查「最近 N 分钟内是否有飞书请求」；若有则执行一次同步检查 | 不依赖改 Gateway 代码，可用 cron + 简单脚本 | 非实时，依赖「最近活跃」的统计方式 |

**建议**：优先考虑 **A**，在实现难度与体验间平衡；若 Gateway 不便改动的环境可采用 **C** 或 **B**。

### 3.3 更新与重启策略

- **更新动作**：在仓库根目录执行 `git fetch origin main`，比较 `origin/main` 与当前 `HEAD`；若有差异则 `git pull origin main`（或 `git reset --hard origin/main`，视策略而定）。
- **是否需要重启**：可根据「变更文件路径」决定，例如：
  - 若 `config/`、`scripts/start-openclaw.sh`、`ecosystem.config.cjs` 等发生变更 → 视为需要重启；
  - 仅 `docs/` 或部分 markdown 变更 → 可不重启，仅更新文件。
- **重启命令**：在仓库根执行 `pm2 restart claw-gateway`，确保使用最新代码。

## 四、功能需求（验收标准）

1. **触发**  
   - 飞书侧有聊天行为时，云端能在约定方式下触发一次「同步检查」（可为请求钩子、Webhook 或定时+活跃判断）。

2. **拉取与比较**  
   - 同步检查时在仓库目录执行 `git fetch origin main`，并正确比较 `origin/main` 与当前分支/HEAD，能判断「是否有新 commit」。

3. **更新**  
   - 若有新 commit，执行拉取（pull 或 reset）使工作区与 `origin/main` 一致；过程可记录日志（谁触发、是否有变更、是否执行了更新）。

4. **按需重启**  
   - 当变更涉及「需重启」的文件或规则时，在更新后自动执行 PM2 重启（`pm2 restart claw-gateway`）；否则仅更新文件。重启策略可配置（如通过环境变量或配置文件列出「触发重启的路径」）。

5. **安全与稳定性**  
   - 触发入口需鉴权（若为 HTTP 接口）或仅在受信环境执行；同一时刻仅允许一个「同步+更新+重启」流程，避免并发拉取/重启；失败时记录日志并可选告警（如飞书消息）。

6. **可观测**  
   - 关键步骤（触发、有/无变更、执行更新、执行重启）有日志可查；可选：在飞书回复中简要提示「已检查 / 已更新 / 已重启」。

---

## 五、非目标（Out of Scope）

- 不要求「每次飞书消息都触发」：可通过节流/防抖或「最近 N 分钟仅执行一次」限制频率。
- 不要求支持除 main 以外的分支自动同步（仅 main 与云端联动）。
- 不要求实现完整的 Git 冲突解决：云端以「与 origin/main 一致」为目标，采用 pull 或 reset 策略即可。
- 不要求本 PRD 包含飞书机器人对话内容本身的设计（仅使用「飞书有请求」作为触发信号）。

## 六、依赖与前置条件

- 云端机器已部署 OpenClaw Gateway（PM2），且能访问 GitHub（`git clone`/`git pull` 所需网络与权限）。
- 仓库在云端以 Git 方式存在，且已配置 `origin` 指向 GitHub 主仓库；建议使用 SSH 或 token，避免交互式认证。
- 若采用 Webhook 方案：需在飞书侧或中间服务配置调用云端 URL；若采用请求钩子：需本仓库或 OpenClaw 支持扩展点/中间件。

## 七、实现建议（概要）

1. **同步执行模块**  
   - 独立脚本（如 `scripts/sync-from-github.sh`）：接收参数（如 `--check-only` / `--pull-and-restart`），内部完成 fetch、比较、pull、判断是否重启、执行 `pm2 restart claw-gateway`。可被 cron、Webhook 或 Gateway 钩子调用。

2. **触发集成**  
   - 若选方案 A：在启动脚本或 Gateway 封装中，在飞书请求路径上增加「节流后的同步检查」调用（如通过子进程或队列异步执行，不阻塞响应）。
   - 若选方案 B：在云端暴露一个带 token 的 HTTP 接口，仅做「触发一次 sync 脚本」；飞书侧配置 Outgoing Webhook 或自建服务调用该 URL。

3. **重启策略配置**  
   - 在仓库或 `.env` 中维护「触发重启的文件路径列表」或规则（如 `config/,scripts/,ecosystem.config.cjs`），sync 脚本根据 `git diff` 结果判断是否执行重启。

4. **日志与可选通知**  
   - 脚本输出与错误写入固定日志文件；可选：在同步/重启完成后通过 OpenClaw 或飞书 API 给指定用户发一条简要状态消息。
